# 八章开发路线与验收标准

每章使用独立分支和 PR。只有验收标准全部满足后，才更新 `README.md` 中的章节状态。

## AI 协作执行计划

角色、任务包、分支隔离和审查细则见 [`ai-collaboration.md`](ai-collaboration.md)。每章默认协作强度如下；如果任务触发更高风险条件，必须升级流程。

| 章节 | 默认流程 | 启用 ZCode 的重点 |
| --- | --- | --- |
| 01 | Cursor 实现，Codex 验收；低风险时可由 Codex 单独实现 | 最大步数、正常终止和无进展循环；作为三工具低风险试运行候选 |
| 02 | Cursor 实现，ZCode 定向审查，Codex 验收 | 状态转换、上下文增长、检查点和中断恢复 |
| 03 | 三角色完整流程 | 工具错误、重试降级、Prompt Injection、SSRF、Python 隔离和来源元数据 |
| 04 | 三角色完整流程 | 记忆裁剪、序列化兼容、恢复、过期与删除 |
| 05 | 教学实验采用双角色；真实数据流水线采用三角色 | 数据泄漏、划分稳定性、轨迹 schema、Loss Mask 和质量统计 |
| 06 | 直觉实验采用双角色；真实训练与奖励采用三角色 | 奖励投机、配置复现、GPU smoke test 和基线对比 |
| 07 | 三角色完整流程 | 并发、取消、超时、恢复、缓存、指标、成本和部署安全 |
| 08 | 三角色完整流程 | 引用支持度、重复来源、不确定性、Judge 版本和 Goodhart 风险 |

前两个功能 PR 是协作试运行期。每个 PR 记录有效缺陷、误报、修复轮数、总耗时和额外成本，并在第 3 章开始前复盘是否调整角色分工。

## 第 1 章：为什么需要 Deep Research

- 运行一个有最大步数限制的 mini ReAct Agent；
- 使用 stub Search 工具完成多步研究；
- 输出动作、观察和最终答案；
- 测试正常终止与最大步数终止。

## 第 2 章：ReAct → IterResearch

- 用显式状态机实现计划、执行、观察和更新；
- 使用演进报告控制上下文增长；
- 支持检查点和中断恢复；
- 对比 ReAct 与 IterResearch 的上下文规模和结果。

## 第 3 章：工具集设计

- 实现统一 Tool 协议和 Tool Registry；
- 提供 Search、Visit、Scholar、Python 的 stub 或真实适配器；
- 覆盖参数校验、超时、重试、缓存、未知工具和来源元数据；
- Python 执行必须隔离。

## 第 4 章：多轮对话与长程记忆

- 区分会话内检查点与跨会话长期记忆；
- 支持滑动窗口、摘要和指代消解；
- 支持按用户、时间和语义检索长期记忆；
- 提供更新、过期和删除机制。

## 第 5 章：训练数据构造

- 定义版本化轨迹 schema；
- 实现种子、轨迹、增强、质检和导出流水线；
- 正确生成 Loss Mask；
- 固定数据划分并生成质量统计。

## 第 6 章：多阶段训练

- 提供零依赖 GRPO 直觉实验；
- 提供可选 SFT/GRPO 配置与最小 GPU smoke test；
- 奖励函数可单独测试；
- 记录基线、训练配置和评估差异。

## 第 7 章：工程化上线

- 提供 API 和流式研究事件；
- 支持并发限制、取消、超时、恢复和缓存；
- 记录延迟、Token、成本、错误率和工具调用次数；
- 提供 Docker 化启动方式和健康检查。

## 第 8 章：评估闭环

- 同时评估最终报告和研究过程；
- 覆盖正确性、完整性、引用、工具效率和成本；
- Judge 提示词、模型和评分标准版本化；
- 评估失败样本可回流为数据改进任务。
